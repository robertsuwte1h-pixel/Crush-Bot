# Telegram Crush Bot with Playwright card generation

A Telegram bot that collects a crush's name from the user, generates a personalized "Do you love me?" HTML card (from an existing template), and handles a deep-link callback when the crush clicks "Yes" — at which point the bot uses Playwright to screenshot a "Said Yes" scrapbook template and sends the resulting image back to the original user. The bot runs on Railway as a polling worker with JSON-file persistence.

**Watch for:** The HTML replacement strategy uses exact string matching against the original template's `fetch` URL, which will silently fail if the template is even slightly reformatted (confirmed). No access control on `/admin` (confirmed). Temp-file race on concurrent requests (likely). Playwright `install-deps` never runs on Railway, risking Chromium launch failure (likely).

## High-level view

The core flow hinges on a string-replacement approach to HTML generation: the bot reads the template, replaces a literal multi-line JavaScript snippet containing a hardcoded Replit URL with a new snippet containing the Telegram deep link. If the template's whitespace or URL changes by a single character, the replacement produces no change and the generated card still points at the dead Replit endpoint — no error raised.

Persistence is a JSON file (`user_data.json`) written and read on every operation with no locking. For a single-process polling bot this is tolerable, but concurrent deep-link arrivals can corrupt the file since `load` + `save` is not atomic.

The `/admin` command has no authorization check — any user who discovers or guesses the command gets full visibility into all stored crush names and Telegram user IDs.

On Railway, `bot.py` runs `playwright install chromium` at startup (installs the browser binary) but never runs `playwright install-deps chromium` (installs OS-level libraries like libnss3, libatk). The `playwright_setup.sh` script exists but is never referenced by any build config. Chromium will likely fail to launch on a fresh Railway deploy.

<details>
<summary>Issues (6)</summary>

1. **Brittle HTML replacement** — The `generate_love_html` function matches the old Yes-button handler as an exact multi-line string. Any change to the template (whitespace, URL update) silently breaks the replacement. Use a regex or DOM manipulation instead.
2. **No admin authorization** — `/admin` exposes all user data (names, IDs, statuses) to any Telegram user. Gate it behind an `ADMIN_USER_ID` environment variable check.
3. **File descriptor leak in `reply_document`** — `open(output_path, "rb")` is passed directly without a context manager; the file handle is never explicitly closed. Wrap in `with` or use `BytesIO`.
4. **Temp HTML race in `generate_said_yes_image`** — A single fixed filename `temp_said_yes.html` is written and read for every request. Concurrent invocations overwrite each other. Use `tempfile.NamedTemporaryFile` or include user_id in the filename.
5. **JSON persistence not atomic** — `load_user_data()` followed by `save_user_data()` in `set_user_entry` has a TOCTOU window. Under concurrent deep-link arrivals, one write can clobber another. Use `filelock` or write-to-temp-then-rename.
6. **Missing `install-deps` on Railway** — `bot.py` installs the Chromium binary at startup but not the OS-level dependencies. Reference `playwright_setup.sh` in a Railway build command or add `install-deps` to the subprocess call.

## Brittle template replacement

The `generate_love_html` function replaces the Yes-button click handler by matching a 7-line JavaScript block character-for-character, including the hardcoded Replit URL:

```python
old_yes_handler = """    yesBtn.addEventListener('click', () => {
        quizSection.style.display = 'none';
        noBtn.style.display = 'none';
        successSection.style.display = 'flex';
        // Notify the sender
        if ('https://51a6c7ef-...') {
            fetch('https://51a6c7ef-...').catch(() => {});
        }
    });"""
```

If the template file is ever reformatted (tabs vs spaces, trailing whitespace, URL update), `str.replace` returns the original string unchanged — no error raised. The generated card would still show the success animation but call a dead Replit endpoint instead of redirecting to the bot. A more robust approach: use `re.sub(r"yesBtn\.addEventListener\('click'.*?\}\);", new_handler, html_content, flags=re.DOTALL)` or inject the deep link URL into a placeholder variable in the template.

## Admin access is open

```python
async def admin(update: Update, context: ContextTypes.DEFAULT_TYPE) -> None:
    user_data = get_user_data()
    ...
```

Any Telegram user who sends `/admin` receives the count of all cards and statuses. The data itself (crush names, user first names) is stored in `user_data.json` and could be enumerated if the admin handler were expanded. At minimum, check `update.message.from_user.id` against a configured admin ID.

## File descriptor leak

In `receive_name`:

```python
await update.message.reply_document(
    document=open(output_path, "rb"),
    ...
)
```

The `open()` call returns a file object that is never explicitly closed. On PyPy or if exceptions occur between `open` and the send, the descriptor leaks. Use:

```python
with open(output_path, "rb") as f:
    await update.message.reply_document(document=f, ...)
```

## Temp file collision in image generation

`generate_said_yes_image` writes to a fixed path `temp_said_yes.html`. If two crushes click "Yes" simultaneously, one request's HTML is overwritten by the other before Playwright captures it. The `finally` block also deletes the file, which could remove another request's in-progress template. Using `tempfile.NamedTemporaryFile(suffix='.html', delete=False)` eliminates the race.

## Desktop deep link fallback gap

`window.location.href` in the generated HTML redirects to `t.me/BotName?start=yes_USERID`. On desktop, if the user doesn't have Telegram installed, the `t.me` link opens in-browser and the bot never receives the deep link. There's no fallback messaging or instruction to open in the Telegram app. This is a UX gap rather than a code bug (possible).

## Playwright system dependencies on Railway

`bot.py` runs at startup:

```python
subprocess.run(["playwright", "install", "chromium"], ...)
```

This installs the Chromium binary but *not* the OS-level shared libraries it depends on (libnss3, libatk-bridge, libgbm, etc.). The separate `playwright_setup.sh` script has both `install` and `install-deps`, but nothing in the `Procfile` or Railway build config invokes it. On a fresh Railway container, Chromium will crash with missing `.so` errors. Either add `install-deps` to the subprocess call in `bot.py`, or configure Railway to run `playwright_setup.sh` during the build phase.
